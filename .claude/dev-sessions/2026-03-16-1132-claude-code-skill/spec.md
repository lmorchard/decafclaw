# Claude Code Skill — Spec

## Goal

Give DecafClaw the ability to delegate coding tasks to Claude Code as a subagent. Users interact with DecafClaw in Mattermost; when coding work is needed, DecafClaw spins up Claude Code sessions that can read, edit, and execute code in specified repositories.

## Use Cases

1. **Bulk work (primary)** — User asks DecafClaw to fix a bug, add a feature, refactor code. DecafClaw delegates to Claude Code, which does the work, then DecafClaw summarizes the results.
2. **Iterative refinement** — After bulk work, user tweaks via follow-up messages. Claude Code session persists, so context is maintained. "Actually use a different approach" or "now add a test for it."
3. **Autonomous (future)** — Heartbeat or scheduled tasks that trigger Claude Code for maintenance. Needs more thought on use cases before implementing.

## Architecture

### Integration via Claude Agent SDK

Uses the `claude-agent-sdk` Python package (successor to `claude-code-sdk`). This provides programmatic access to Claude Code with:
- Async streaming API
- Persistent sessions (`ClaudeSDKClient`)
- Permission callbacks (`can_use_tool`)
- Cost tracking (`ResultMessage.total_cost_usd`)
- Working directory, model, and budget controls

### Authentication

`ANTHROPIC_API_KEY` environment variable in `.env`. The SDK reads it automatically.

## Tool Surface

Four tools exposed when the skill is activated:

### `claude_code_start`
Start a new Claude Code session.

**Parameters:**
- `cwd` (required, string) — working directory / project path
- `description` (optional, string) — what this session is for (helps with session listing)
- `model` (optional, string) — override the default Claude model
- `budget_usd` (optional, float) — per-session budget override (up to configured max)

**Behavior:**
- Creates a new `ClaudeSDKClient` session
- One active session per working directory — if a session already exists for that `cwd`, return an error with guidance to use `claude_code_send` or `claude_code_stop` first
- Returns a session ID for subsequent tool calls

### `claude_code_send`
Send a prompt to an existing session.

**Parameters:**
- `session_id` (required, string) — which session to send to
- `prompt` (required, string) — the message/task for Claude Code

**Behavior:**
- Sends the prompt to the session
- Streams messages, routing permission requests through Mattermost confirmation flow
- Logs full output to disk
- Returns a summary of what Claude Code did (generated by DecafClaw's LLM from the full output)
- If the session has expired, return an error indicating expiration — agent must start a new session

### `claude_code_stop`
End a session and clean up.

**Parameters:**
- `session_id` (required, string)

**Behavior:**
- Closes the `ClaudeSDKClient`
- Session is no longer usable
- Returns confirmation with final cost summary

### `claude_code_sessions`
List active sessions.

**Parameters:** none

**Returns:** list of active sessions with session ID, working directory, description, age, and idle time.

## Session Management

### Lifecycle
- Sessions are created via `claude_code_start` and persist across multiple `claude_code_send` calls
- Sessions maintain full conversation history within the Claude Code context
- One active session per working directory (prevents conflicting edits)
- Multiple sessions across different working directories are fine

### Idle Expiration
- Sessions auto-expire after a configurable idle timeout (default: 30 minutes)
- Configurable via `CLAUDE_CODE_SESSION_TIMEOUT` env var (seconds or interval string)
- When an expired session is accessed via `claude_code_send`, the agent gets an error indicating the session expired and must explicitly start a new one
- This is intentional: forces the agent to re-establish context and restate the problem to the new Claude Code instance
- Expiration check runs lazily (on next access), not via background timer

### Cleanup
- `claude_code_stop` explicitly closes a session
- Skill deactivation should close all active sessions (may require extending the skill system's lifecycle hooks — see Design Notes)

## Permission Model

### Confirmation Flow
Uses the SDK's `can_use_tool` callback to intercept tool execution:

1. Claude Code wants to execute a tool (e.g. `Bash("npm install")`)
2. `can_use_tool` callback fires
3. Check the Claude Code allowlist — if it matches, auto-approve
4. If not, publish a confirmation event through DecafClaw's event bus
5. Mattermost posts the confirmation message in the conversation thread
6. User reacts (approve/deny/always)
7. Response flows back to the `can_use_tool` callback
8. Claude Code proceeds or skips

This reuses DecafClaw's existing `request_confirmation()` helper and Mattermost reaction-polling pattern.

**Key concern:** The `can_use_tool` callback must async-block inside the SDK's event loop until the user responds (or timeout). This is the same pattern as our shell tool confirmation, but running inside the SDK's context. Needs prototyping to verify it works cleanly.

### Allowlist
- Separate file: `claude_code_allow_patterns.json` in the agent path (admin-managed, outside workspace)
- Same pattern as `shell_allow_patterns.json` — glob patterns, admin-managed
- "Always" reactions add to this allowlist
- Read-only tools (`Read`, `Glob`, `Grep`) should probably be auto-approved by default

### Permission Modes
The SDK supports `acceptEdits` (auto-approve file ops) and `bypassPermissions` (auto-approve everything). We may want to expose these as a config option for trusted environments, but default to the confirmation flow described above.

## Output & Logging

### Full Logs
- All session output logged to `workspace/claude-code-logs/{session-id}.jsonl`
- Each line is a JSON object representing a message from the SDK stream
- Includes reasoning, tool calls, tool results, cost info
- Agent can read these via workspace tools if the user asks for details

### Conversation Summary
- After each `claude_code_send`, DecafClaw's LLM receives a summary (not the full output) to save tokens
- Summary should include: what was done, files changed, any errors, cost
- Full log path included so the agent can offer to share details

## Configuration

All via environment variables in `.env`:

| Variable | Default | Description |
|----------|---------|-------------|
| `ANTHROPIC_API_KEY` | (required) | API key for Claude Code |
| `CLAUDE_CODE_MODEL` | (SDK default) | Default Claude model for sessions |
| `CLAUDE_CODE_BUDGET_DEFAULT` | `2.00` | Default per-session budget in USD |
| `CLAUDE_CODE_BUDGET_MAX` | `10.00` | Maximum per-session budget in USD |
| `CLAUDE_CODE_SESSION_TIMEOUT` | `30m` | Idle timeout before session expires |

## Design Notes

### Skill System Evolution
The existing skill system (`init(config)` + `TOOLS` + `TOOL_DEFINITIONS`) may need extensions for this skill:

- **Async init** — `init()` may need to be async (already supported via `_call_init`)
- **Shutdown hook** — no `shutdown()` or `cleanup()` hook exists today. This skill needs to close SDK clients on deactivation. We may need to add a `shutdown(config)` lifecycle hook to the skill system.
- **State management** — the tabstack skill uses a module-level `_client` global. This skill will need more state (multiple sessions, expiration timers). Consider whether this should live in module globals or on the context.

These are areas where we should evolve the skill contracts as needed rather than over-designing upfront.

### Prototype-First Areas
- The `can_use_tool` → Mattermost confirmation async bridge is the riskiest part. Prototype this first.
- Session lifecycle management (idle expiration, cleanup) is the most complex state management. Start simple (lazy expiration check), add sophistication later.
- Output summarization quality depends on how verbose the SDK output is. May need tuning.

### SDK Package Name
Research suggests the package was renamed from `claude-code-sdk` to `claude-agent-sdk`. Verify the actual package name and API before implementation — install it and check the imports.

## Out of Scope (for now)

- Autonomous/heartbeat-triggered Claude Code tasks
- Multi-agent coordination (multiple Claude Code sessions working on the same repo)
- Image/screenshot support in prompts
- Claude Code MCP server integration (Claude Code can itself use MCP servers — not needed initially)
