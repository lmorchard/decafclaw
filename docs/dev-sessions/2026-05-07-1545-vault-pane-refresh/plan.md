# Vault left-pane live refresh — Implementation Plan

**Goal:** Push a `vault_changed` WebSocket message to all connected clients on every vault mutation (11 paths: 6 agent tools + 5 REST handlers), so the sidebar auto-refreshes via the existing REST endpoints.

**Approach:** Three vertical slices. Phase 1 lights up the full end-to-end pipeline (helper + wire type + per-connection subscriber + client dispatch + sidebar listener) wired through `tool_vault_write` only — proves the whole pipeline before extending coverage. Phase 2 extends to the remaining tool surface. Phase 3 extends to REST handlers and removes the now-redundant `vault-page-deleted` DOM event in the same commit.

**Tech stack:** Python 3 (asyncio, EventBus), Lit-element JS, JSON-driven WS message types, pytest.

**Architectural note:** the existing `_make_notification_forwarder` at `src/decafclaw/web/websocket.py:710-732` is the precedent — each WebSocket connection self-subscribes to the global EventBus and filters by `event["type"]`. There's no centralized "broadcast to all" manager; "broadcast" emerges naturally because every connection has its own subscriber. We add a `_make_vault_change_forwarder` next to it.

```python
# src/decafclaw/skills/vault/_events.py
"""EventBus publisher for vault mutations.

Vault tools and REST handlers call publish_vault_changed() after a successful
disk operation. The web UI's per-connection subscriber (in web/websocket.py)
fans the event out as a `vault_changed` WebSocket message to every connected
client. Mirrors the notification forwarder pattern.

Fail-open: a publish failure must never break the calling tool or handler.
"""

import logging
from pathlib import Path

log = logging.getLogger(__name__)

VAULT_CHANGED_EVENT_TYPE = "vault_changed"

# Allowed `kind` values. Advisory only — clients always re-fetch regardless.
KIND_CREATE = "create"
KIND_UPDATE = "update"
KIND_DELETE = "delete"
KIND_RENAME = "rename"
KIND_MOVE = "move"
KIND_SECTION = "section"
KIND_JOURNAL = "journal"


async def publish_vault_changed(
    event_bus,
    config,
    kind: str,
    path: str | Path | None = None,
) -> None:
    """Publish a vault_changed event for the web UI to broadcast.

    Args:
        event_bus: The shared EventBus (ctx.event_bus or
            request.app.state.event_bus).
        config: The runtime config; used to resolve `path` against
            `config.vault_root` for relative-path normalization.
        kind: One of KIND_CREATE/UPDATE/DELETE/RENAME/MOVE/SECTION/JOURNAL.
        path: The changed file's path (str, Path, or None). Absolute paths
            are normalized to vault-relative; None becomes "" in the
            payload (used for multi-file ops where there's no single path).

    Fail-open: any exception is logged at debug level and swallowed.
    """
    if event_bus is None:
        return
    rel = ""
    if path is not None:
        try:
            p = Path(path)
            if p.is_absolute():
                rel = str(p.resolve().relative_to(config.vault_root.resolve()))
            else:
                rel = str(p)
            rel = rel.replace("\\", "/")
        except (ValueError, OSError) as exc:
            log.debug("vault_changed path normalize failed for %r: %s", path, exc)
            rel = ""
    try:
        await event_bus.publish({
            "type": VAULT_CHANGED_EVENT_TYPE,
            "kind": kind,
            "path": rel,
        })
    except Exception as exc:
        log.debug("vault_changed publish failed: %s", exc)
```

```python
# src/decafclaw/web/websocket.py — add next to _make_notification_forwarder
def _make_vault_change_forwarder(ws_send):
    """Forward `vault_changed` events from the global bus to a single socket.

    Mirrors `_make_notification_forwarder`: filter by event type, send the
    matching wire shape to the client. The bus dispatches every event to
    every subscriber, so per-event filtering is required.
    """
    async def _forward(event: dict):
        if event.get("type") != "vault_changed":
            return
        await ws_send({
            "type": WSMessageType.VAULT_CHANGED,
            "path": event.get("path", ""),
            "kind": event.get("kind", ""),
        })
    return _forward
```

## What this plan does NOT do

(Mirror of `spec.md`'s "What we're NOT doing" — for plan reviewer reference.)

- No debouncing or throttling.
- No conversation scoping (vault is global).
- No per-user scoping (single-user app today).
- No incremental tree mutation on the client.
- No new data in the message beyond `path` and `kind`.
- No coalescing in the WebSocket subscriber.
- No live-reload of the in-editor open page (`wiki-page.js`).
- No retroactive event publication on server start.
- No persistence of `vault_changed` events.
- No new auth surface (relies on existing WS auth).

## Plan self-review

- **Spec coverage:** every spec requirement maps to a phase:
  - Wire type + helper + subscriber + client dispatch + sidebar listener → Phase 1.
  - 6 tool mutations → Phase 1 (`vault_write`) + Phase 2 (other 5).
  - 5 REST mutations → Phase 3.
  - Remove `vault-page-deleted` DOM event → Phase 3.
  - Tests at every layer → Phases 1–3.
- **Type consistency:** `publish_vault_changed(event_bus, config, kind, path)` signature is identical at every call site. `KIND_*` constants imported from `_events.py` everywhere. Wire payload `{type, kind, path}` consistent in spec, JSON, JS dispatch, Python forwarder.
- **No placeholders:** every phase has concrete code snippets, file paths, line refs, and test outlines. Implementation-time lookups are explicitly noted (e.g., "look at the existing `finally` block when implementing"); they aren't TBDs because the surrounding context already pins the shape.
- **Vertical slice principle:** Phase 1 alone delivers a working feature for `vault_write`. Phase 2 expands coverage. Phase 3 closes the REST gap and removes redundant code. Each phase is independently revertable and useful if the later phases stall.
