# Interactive Buttons — Spec

## Goal

Add a lightweight HTTP server to DecafClaw and use Mattermost's interactive message buttons for tool confirmation UI. This replaces emoji-reaction polling as the primary confirmation mechanism while keeping emoji as a fallback. The HTTP server infrastructure is also groundwork for a future web-based gateway UI.

## Two Layers

### 1. HTTP Server Infrastructure

A Starlette ASGI server running alongside the existing websocket client in the same async event loop.

**Configuration:**

| Variable | Default | Description |
|----------|---------|-------------|
| `HTTP_ENABLED` | `false` | Start the HTTP server |
| `HTTP_HOST` | `0.0.0.0` | Bind address |
| `HTTP_PORT` | `18880` | Listen port |
| `HTTP_SECRET` | (required if enabled) | Shared secret for callback verification |
| `HTTP_BASE_URL` | (auto-detected) | External URL for callback URLs, e.g. `http://decafclaw.velociraptor-bun.ts.net:18880`. Auto-detected from host/port if not set. |

**Lifecycle:**
- Starts in `MattermostClient.run()` if `HTTP_ENABLED=true`
- Runs as an asyncio task alongside the websocket listener
- Shuts down gracefully with the rest of the bot

**Security:**
- Shared secret (`HTTP_SECRET`) included in callback URLs as a query parameter
- Callbacks reject requests without a valid secret
- Secret should be a random string, generated by the admin

### 2. Interactive Button Confirmation UI

Replace emoji-polling confirmation messages with Mattermost interactive message buttons.

**Button layouts:**

Shell tool:
- **Approve** (style: primary) — approve this command
- **Deny** (style: danger) — deny this command
- **Allow Pattern** (style: default) — approve and add the suggested pattern to the allowlist

All other tools (skill activation, claude_code_send, etc.):
- **Approve** (style: primary) — approve this time
- **Deny** (style: danger) — deny
- **Always** (style: default) — approve and remember

**Flow:**

1. Tool requests confirmation → `request_confirmation()` publishes event
2. Mattermost subscriber receives event, posts a message with:
   - Text description of what needs approval
   - Interactive buttons (via `attachments[].actions[]`)
   - Emoji instructions as fallback text (if emoji not disabled)
3. User clicks a button → Mattermost POSTs to `http://<decafclaw>:18880/actions/confirm?secret=<secret>`
4. HTTP handler:
   - Verifies secret
   - Publishes `tool_confirm_response` event on the event bus
   - Returns response that updates the post (removes buttons, shows result)
5. `request_confirmation()` receives the event and returns to the tool

**Parallel paths:** Both buttons and emoji reactions can resolve the same confirmation. Whichever arrives first wins — the other is ignored (the asyncio.Event is already set).

## Confirmation Message Format

Current (emoji only):
```
🚨 **Confirm shell:**
` ``
ls -la
` ``
React: 👍 approve | 👎 deny | ✅ always | 📓 allow `ls *`
```

New (buttons + emoji):
```
🚨 **Confirm shell:**
` ``
ls -la
` ``
[Approve] [Deny] [Allow Pattern: ls *]

React: 👍 approve | 👎 deny | 📓 allow `ls *`
```

The emoji line is shown unless `MATTERMOST_DISABLE_EMOJI_CONFIRMS=true`.

**After button click, message updates to:**
```
🚨 **Confirm shell:**
` ``
ls -la
` ``
**Result:** ✅ Approved by @lmorchard
```

Buttons are removed in the update response. No separate API call needed.

## HTTP Routes

### `POST /actions/confirm`

Receives Mattermost interactive message button callbacks.

**Query params:**
- `secret` — must match `HTTP_SECRET`

**Request body (from Mattermost):**
```json
{
  "user_id": "rd49ehbqyjytddasoownkuqrxe",
  "post_id": "gqrnh3675jfxzftnjyjfe4udeh",
  "channel_id": "j6j53p28k6urx15fpcgsr20psq",
  "team_id": "5xxzt146eax4tul69409opqjlf",
  "context": {
    "action": "approve",
    "context_id": "abc123",
    "tool": "shell",
    "suggested_pattern": "ls *"
  }
}
```

**Response (updates the post):**
```json
{
  "update": {
    "message": "🚨 **Confirm shell:**\n```\nls -la\n```\n\n**Result:** ✅ Approved by @username",
    "props": {
      "attachments": []
    }
  }
}
```

### `GET /health`

Simple health check for monitoring. Returns `{"status": "ok"}`.

## Integration Points

### Confirmation message posting (`_subscribe_progress` / `ConversationDisplay`)

Currently posts a text message with emoji instructions, then starts a `_poll_confirmation` task. Changes:

- If HTTP server is running, include button attachments in the message
- Still start `_poll_confirmation` for emoji (unless disabled)
- Either path (button click or emoji reaction) resolves the confirmation

### `_poll_confirmation`

Keep as-is — it polls for emoji reactions. If buttons resolve first, the poll task is cancelled in the finally block (already happens).

### HTTP callback handler

New code. Receives button clicks, looks up the pending confirmation by context_id + tool, publishes the event. The handler is stateless — it just bridges the HTTP request to the event bus.

### `request_confirmation()` helper

No changes needed — it already works via event bus pub/sub. The HTTP handler just publishes the same `tool_confirm_response` event that emoji polling publishes.

## Design Notes

### Why Starlette

- Lightweight ASGI framework, async-native
- Real routing and middleware for when we build the web gateway UI
- `uvicorn` as the ASGI server runs in the same event loop

### Event bus as the bridge

The HTTP handler doesn't need direct access to the pending confirmation state. It publishes an event, and `request_confirmation()` picks it up via its subscriber. This keeps the HTTP layer decoupled from the tool confirmation logic.

### Race condition handling

Both emoji and button can resolve the same confirmation. `request_confirmation()` uses `asyncio.Event.set()` which is idempotent — the second resolution is a no-op. The `_poll_confirmation` task is cancelled in the finally block after the confirmation resolves.

## Out of Scope

- Web gateway UI (future session — uses the same HTTP server)
- Authentication beyond shared secret (e.g., user tokens, OAuth)
- HTTPS termination (handled by reverse proxy in production)
- Interactive dialogs (Mattermost supports modal forms — not needed for confirmations)
